Codex Windows 桌面版代理超时问题解决记录

记录日期：2026-05-17；环境：Windows 桌面版 Codex；问题表现：新建对话时出现子进程退出等待超时。

Windows Codex Desktop HTTP_PROXY 本地代理 已解决

问题现象

在 Codex Windows 桌面版中新建对话时，界面会出现连接异常，最终报错类似：

timeout waiting for child process to exit

这个问题只在新对话或子进程启动链路中更明显，已有会话未必总是复现。

排查结论

网上同类案例指向代理环境：Codex 桌面端启动的新子进程不一定能稳定读取系统代理或代理工具的 PAC/系统代理设置，因此网络请求可能没有走本地代理。

本机代理工具 HTTP 端口 59527
本机代理工具 SOCKS5 端口 59526
推荐给 Codex 使用 HTTP 代理：http://127.0.0.1:59527
不优先使用 SOCKS5 端口，因为部分 Electron/Node 场景对 HTTP 代理变量更稳。

实际生效的修复

在 PowerShell 中设置用户级代理环境变量，让新启动的 Codex 进程继承代理配置：

setx HTTP_PROXY  "http://127.0.0.1:59527"
setx HTTPS_PROXY "http://127.0.0.1:59527"
setx ALL_PROXY   "http://127.0.0.1:59527"
执行以上三行后，问题似乎已经解决。关键是之后要完全退出并重新打开 Codex，让新进程继承这些环境变量。

建议补充项

如果后续遇到本地接口、localhost 或内网访问被错误代理，可以额外设置：

setx NO_PROXY "localhost,127.0.0.1,::1"

如果将来代理工具端口变化，需要把上面三项中的 59527 改成新的 HTTP 端口。

复现时处理顺序

  1. 确认代理工具的 HTTP 代理已启用，端口仍是 59527。
  2. 确认 PowerShell 新窗口中能看到环境变量：Get-ChildItem Env:HTTP_PROXY,Env:HTTPS_PROXY,Env:ALL_PROXY。
  3. 完全退出 Codex 桌面版，再重新打开。
  4. 仍然失败时，尝试开启代理工具的 Tunnel/TUN 模式，让流量在网络层被接管。
  5. 也可以使用 Codex Proxy Switcher 之类启动器，把代理填为 http://127.0.0.1:59527。
注意：setx 修改的是未来新进程的用户环境变量，不会改变当前已打开程序的环境。设置后必须重启 Codex 才会生效。
